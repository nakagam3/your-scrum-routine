# サイクルアーキテクチャ

your-scrum-routine の全体設計図。
PDCA フラクタル構造に基づき、各スケールの各フェーズを担当するスキルをマッピングする。

## 原点: retrospective が解いた痛み

```
表層: 「日報が書けない・続かない」
  ↓
中層: 「今日やったことが分からない」「明日のGoalが思いつかない」
  ↓
深層: 「ふりかえりができる状態にない」
  ↓
本質: ゴールと現在地の Gap 確認 → 次の一歩を決める、が回っていない
```

retrospective は「ゴール」と「現在地」の確認を仕組み化し、Gap に対する次の一歩を提案することで最小コストで回せるようにした。

## 境界原則

スキルの境界は **タイミング × 責務** で引く。タスクの種類（凡事/プロジェクト/個人依頼）では分けない。
Scrum の Sprint Backlog にストーリーもバグもタスクも区別なく載るのと同じ。

## PDCA フラクタル全体マップ

同じ C→A→P 構造が複数のスケールで相似形を成す。

```
スケール    P (Plan)             D (Do)        C (Check)           A (Act)
─────────────────────────────────────────────────────────────────────────────
Q (四半期)  Q目標・MS定義        週→日の実行   Q振り返り            次Qの目標調整
            [manual]                           [manual]            [manual]

月          月次MS確認            週→日の実行   月末振り返り         MS調整
            [manual]                           [manual]            [manual]

週          MS選択               日の実行       週次ふりかえり       来週の調整
            +全タスク時間軸配置                  +タスク消化率確認
            [#3 未実装]                         [#3 未実装]          [#3 未実装]

日          朝の確定              実行           データ収集・振り返り  明日のゴール提案
            [#1 未実装]                         [retrospective]      [retrospective]

着手        最初の一手を決める    作業            ─                   ─
            [#2 未実装]
```

### 凡例

- `[retrospective]`: 稼働中スキル
- `[manual]`: ユーザーが手動で product-vision/ を編集
- `[#1]` `[#2]` `[#3]`: GitHub Issue 番号（未実装）

### 週次 P フェーズのタスクソース

#3 の P フェーズは全タスクソースを同一フローで時間軸に配置する。

```
タスクソース             取得手段                     備考
──────────────────────────────────────────────────────────────
Google Tasks「凡事」     gws CLI (tasks list)         bonji 設計書から吸収
Jira 自分のアサイン      Atlassian MCP                要検証: API で引けるか
sprint-log 未達ゴール    sprint-log/*.yml 読み取り     retrospective が書き出し済み
今週の MS               マイルストーン.yml 読み取り    product-vision（読み取り専用）
長期物の分割枠           Google Tasks or 手動入力      取り込み方法は要検討
```

時間軸への射影: gcal_find_my_free_time → gcal_create_event。
凡事もプロジェクトタスクも同一フローで枠に詰める（分岐ゼロ）。

## データストア構造: 経営の3層

自分株式会社の経営に必要な情報を3層で整理する。

```
経営の「上」= 目指す場所（product-vision/）
  輝かしい未来.md        ビジョン（3年）
  目指す姿.md            方向性（半年）
  今Q目標.yml            中期計画（Q）
  マイルストーン.yml      実行計画（週次MS）

経営の「真ん中」= 判断基準（インセプションデッキ）   ← #4-#7 で設計予定
  やらないことリスト      NOT List: やらない事業の宣言 (#4)
  夜も眠れない問題        Risks: 事業リスク台帳 (#5)
  トレードオフ・スライダー  What's going to give?: 経営方針 (#6)
  何がどれだけ必要なのか   What it's going to take: 経営資源 (#7)

経営の「下」= 実行と記録
  sprint-log/            日次記録（計画・実績・照合）
  increments/            セッションサマリ
  config.yml             インフラ設定（サービス接続）
```

「上」と「下」は既にある。「真ん中」が空白。
置き場（product-vision/ に含めるか新ディレクトリにするか）は #4-#7 の設計時に決定する。

## ギャップ分析: 見落とし候補

PDCA フラクタルの全スケール・全フェーズを走査し、カバーされていない箇所を洗い出す。

### G1: Q・月スケールの P と A が manual のまま

product-vision/ の編集（Q目標設定、MS 定義、MS 調整）は手動。Q・月スケールの C（確認）も手動運用。

**痛みがあるか?**: Q目標・MS を書く頻度は低い（Q初/月初）。頻度が低い作業は自動化の ROI が出にくい。ただし「MS の調整判断」は週次ふりかえり (#3) の A フェーズで発生するため、#3 が実装されれば部分的にカバーされる可能性がある。

→ **判定保留。#3 を使ってみてから痛みが残るか確認する。**

### G2: 日中の割り込みハンドリング

朝にゴールを確定しても、日中に割り込みが来たとき「今日のゴールを変えるか/守るか」を判断する仕組みがない。

**痛みがあるか?**: 割り込み判断は即時性が求められる。スキルを起動して相談するより、自分で判断するほうが速い場面が多い。ただし「割り込みが来たこと」と「ゴールを変更したこと」の記録がないと、夕方の retrospective で「なぜゴールが達成できなかったか」の分析精度が落ちる。

→ **スキル化の対象ではなく、sprint-log のスキーマで「日中の変更ログ」を持てるかの設計問題。#1 の設計時に検討する。**

### G3: タスク全体像の可視化（バックログ層）

タスクソースが分散（Jira、Google Tasks、個人依頼、長期物）しており、P フェーズの入力品質が担保されない。

**痛みがあるか?**: ある。凡事を忘れる、長期タスクが締切ギリギリになる、という実害が出ている。

→ **#3 の P フェーズ（全タスクの時間軸配置）で解決する設計。独立スキルとしては不要。**
→ ただし「新規タスクの取り込み」（受信→即カレンダー）は運用ルールであり、スキル化の対象外。**

### G4: 成果の可視化（Sprint Review 相当）

一人スクラムで Sprint Review は必要か。チームへの共有義務はないが、「自分が何を成し遂げたか」を定期的に可視化することは達成感とモチベーションに効く。

**痛みがあるか?**: 現時点では不明。週次ふりかえり (#3) の C フェーズが部分的にこの役割を担う可能性がある。

→ **#3 に包含される可能性が高い。独立スキルとしては不要。**

## まとめ: 現時点の優先順位

### スキル（PDCA プロセス）

| 優先度 | 対象 | 根拠 |
|--------|------|------|
| 最優先 | #1 朝の確定エンジン | 代替なし。サイクルの最大の空白 |
| 高     | #3 週次ルーティン | 日次→週次の接続 + 全タスク時間軸配置（bonji 吸収）。G1, G3, G4 もカバー |
| 要検討 | #2 着手変換 | dp が部分代替。#1 に包含するか独立するか設計時に判断。#3 の P フェーズ稼働後に痛みが残るか再評価 |
| 保留   | G1 (Q/月の P&A) | #3 実装後に再評価 |
| 保留   | G2 (割り込み) | スキルではなくスキーマ設計の問題 |
| 解決済 | G3 (タスク全体像) | #3 の P フェーズに吸収 |
| 解決済 | G4 (成果可視化) | #3 の C フェーズに包含 |

### インセプションデッキ（判断基準）

| 優先度 | 対象 | 根拠 |
|--------|------|------|
| 高     | #4 やらないことリスト | スコープ膨張の防波堤。全 P フェーズの判断基準 |
| 高     | #5 夜も眠れない問題 | リスクの可視化。retrospective の検査観点 |
| 中→高  | #6 トレードオフ・スライダー | 経営方針。スキルが増えるほど効く |
| 低     | #7 何がどれだけ必要なのか | 設計上重要だが痛みは低い |
